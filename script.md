## 1. 부하 테스트란?

**부하 테스트(Load Test)** 는

> 시스템이 **예상되는 트래픽을 안정적으로 처리할 수 있는지 검증하는 테스트**이다.
> 

단순히 "최대 몇 명까지 버틴다"를 보는 것이 아니라

→ **서비스가 실제 환경에서 문제없이 동작하는 범위를 확인하는 것**이 핵심이다.

---

## 2. 부하 테스트를 하는 이유

서비스를 운영하면서 가장 큰 리스크 중 하나는

→ **트래픽 급증으로 인한 서버 장애**이다.

예를 들어

- 이벤트 오픈
- 특정 시간대 트래픽 집중
- 마케팅 캠페인
- 티켓팅 폭주
- 대학 수강신청

이런 상황에서 서버가 버티지 못하면

`트래픽 급증 → 서버 장애 → 서비스 중단 → 사용자 이탈 → 신뢰도 하락`으로 이어진다.

### 우리 주변의 실제 사례

### 1) 대학 수강신청 — 매 학기 반복되는 서버 다운

수강신청 시즌이 되면 학생들 사이에서 "서버 터졌다"는 말이 일상이다.

- **명지대 (2020년 8월)**: 수강신청 당일 오전 10시 38분, **L4 스위치 오작동으로 인한 과부하**로 서비스 중단. 수강신청 일정이 다음 날로 연기됨. 문제는 "신청은 막혔지만 삭제는 가능"한 일관성 없는 상태가 발생하면서, 일부 학생들의 수강내역이 비정상적으로 삭제되는 부수적 문제까지 발생.
- **이화여대**: 수강변경 마지막날에 서버가 두 번 연속 다운. 변경기간을 자정까지 연장했으나 그마저도 다시 다운.
- **건국대**: 한 학기 동안 4번의 수강신청 중 문제없던 적이 단 한 번. 학생들이 "정상적으로 수강신청 해본 적이 없다"고 증언할 정도.

> 💡 학교 IT팀은 매 학기 **서버 증설 + 장바구니 제도 도입 + 학년별 분산** 같은 방식으로 대응 중이지만, 본질적으로 이는 모두 **"피크 트래픽 추정"**의 문제이다. 부하 테스트로 한계를 미리 파악하지 못하면, 실제 D-Day에서야 한계점이 드러난다.
> 

### 2) 콘서트·뮤지컬 티켓팅 — 예매 시작 30초가 결정짓는 전쟁

"피켓팅(피 튀기는 티켓팅)"이라는 말이 생길 정도로, 인기 공연 티켓팅은 시작과 동시에 서버에 엄청난 부하가 몰린다.

- **인터파크(NOL 티켓)**: 예매 오픈 직후 동시 접속 폭주로 인해 한 계정 다중 접속을 차단해야 할 정도. 대기열 시스템(`3000 → 2500 → 2000`)으로 트래픽을 인위적으로 분산시킨다.
- **예스24 티켓**: 대기열 없이 한꺼번에 입장시키는 구조여서 **로그인 풀림, 빈 화면, 예매 버튼 사라짐** 등 부하로 인한 다양한 버그가 발생. 이용자들 사이에서는 "예사 티켓팅은 운빨"이라는 말이 정설이 됨.

> 💡 핵심은 **트래픽 분산 전략(대기열, 좌석 잠금 시간, 결제 홀딩)** 이 무너지면 단순히 느려지는 게 아니라 **데이터 정합성까지 깨진다**는 점이다.
> 

### 3) 예스24 서비스 마비 (2025년 6월)

- 서비스가 새벽 4시경 갑자기 전면 마비. 도서 주문, 티켓 예매, eBook 등 **모든 온라인 서비스가 10일간 중단**.
- 도서 주문 취소, 티켓 예매 일정 조정, 팬사인회 응모 취소 등 **막대한 사용자 피해** 발생.
- 이 사례는 부하 문제는 아니었지만(랜섬웨어), **시스템이 멈추면 어떤 도미노 효과가 발생하는지**를 보여주는 대표적 사례.

---

이런 상황들의 공통점은 단 하나다.

> 사전에 **"우리 시스템이 어디까지 버틸 수 있는가"** 를 측정하지 않은 것.
> 

그래서 우리는 배포 전에 반드시 확인해야 한다.

> "우리 서버는 어느 정도 트래픽까지 안정적으로 처리할 수 있는가?"
> 

---

### 💰 왜 그냥 서버를 크게 쓰지 않을까?

이론적으로는 가능하지만 현실에서는 **비용 문제가 발생한다.**

- 인프라 비용은 트래픽 대비 계속 증가
- 피크 시간 기준으로 과하게 구성하면 대부분 시간은 낭비
- 오토스케일링 기준점도 결국 측정값이 있어야 설정 가능

따라서 목표는

> 무조건 큰 서버 → ❌
> 
> 
> **최소 비용으로 안정적인 성능 확보 → ✅**
> 

→ 부하 테스트는

**"터지지 않는 최소 비용 구조"를 찾기 위한 과정**이다.

## 3. 테스트 유형 비교

부하 테스트는 한 종류가 아니다. 목적에 따라 4가지로 나뉜다.

| 종류 | 목적 | 활용 시점 |
| --- | --- | --- |
| **Load Test** (부하 테스트) | 예상 트래픽에서 안정적으로 동작하는지 | 🎯 이번 스터디의 핵심 |
| **Stress Test** (스트레스 테스트) | 한계치를 초과했을 때 시스템이 어떻게 무너지는지 | 내구 한계 파악 |
| **Spike Test** (스파이크 테스트) | 갑작스러운 트래픽 폭증에 버티는지 | 순간 급증 대응 확인 |
| **Soak Test** (내구성 테스트) | 장시간 운영 시 메모리 누수·성능 저하 발견 | 장기 안정성 검증 |

> 💡 k6는 이 모든 테스트 유형을 `options` 시나리오 코드로 표현할 수 있다.
> 

---

## 4. VU(Virtual User) — 가상 사용자

지표를 이해하기 전에 VU 개념이 먼저 필요하다.

> **VU = 사용자 행동을 반복 실행하는 시뮬레이션된 사용자 1명**
> 
> 
> VU 50명 = 최대 50개의 사용자 시나리오가 동시에 실행되는 상황
> 

k6에서는 코드로 이렇게 표현한다.

```jsx
export const options = {
  vus: 50,         // 가상 사용자 50명
  duration: '30s'  // 30초 동안 유지
};
```

## 5. 부하 테스트 핵심 지표 (3가지)

### 5-1. 처리량 (Throughput, TPS)

- 초당 처리 가능한 요청 수
- 단위: TPS (Transactions Per Second)

### TPS 계산 공식

```
TPS ≈ VU 수 ÷ (응답시간 + 대기시간)

예) VU=100, 응답시간=200ms, 대기시간=800ms
→ TPS ≈ 100 ÷ 1.0 = 100 TPS
```

→ 실제 TPS는 VU 수, 응답시간, 대기시간, 시나리오 구조에 함께 영향을 받는다.

### 변화 패턴

- 가상 사용자(VU)가 증가하면 TPS도 증가
- 하지만 어느 순간 증가가 멈춘다
- 이 지점이 → **시스템의 최대 처리 용량 (Capacity)**

---

### 5-2. 지연 시간 (Latency)

- 요청 → 응답까지 걸리는 시간

### ⚠️ 중요 포인트

평균값뿐 아니라 **P95, P99**를 함께 봐야 한다.

실제 사용자 경험은 **느린 요청이 좌우**하기 때문이다.

### P95 / P99란?

| 지표 | 의미 | 활용 |
| --- | --- | --- |
| **P95** | 100개의 요청 중 95개는 이 시간 이하에 응답 | "대부분의 사용자 경험" — 서비스 체감 성능 측정 |
| **P99** | 100개의 요청 중 99개는 이 시간 이하에 응답 | "최악에 가까운 경험" — 병목 구간·장애 직전 감지 |

### 평균의 함정

평균만 보면 안 되는 이유.

먼저 일상적인 예시로 보면 더 쉽다.

직원 9명이 각각 연봉 3천만 원을 받고, CEO 1명이 30억 원을 번다고 해보자.

이 회사의 평균 연봉은 약 3억 원이 된다.

그런데 대표가 밖에 나가서 "우리 회사 평균 연봉이 3억입니다"라고 홍보하고 다니면, 직원들 입장에서는 열받을 수밖에 없다.

대다수 직원의 실제 상황은 3천만 원인데, 평균 하나가 그 현실을 가려버리기 때문이다.

Latency도 똑같다.

```
요청 99개 → 100ms
요청  1개 → 3,000ms
```

→ 평균 = 약 130ms

겉보기에는 빠른 서비스처럼 보이지만, 일부 사용자는 매우 느린 경험을 한다.

→ **평균은 "전체적인 속도"를 보여주지만, P95·P99는 "사용자가 실제로 겪는 느림"을 보여준다.**

### 부하 증가 시 변화 패턴

| 구간 | 평균 | P95 | P99 |
| --- | --- | --- | --- |
| ① 초기 | 안정 | 안정 | 안정 |
| ② 임계점 도달 | 거의 유지 | 서서히 ↑ | **급격히 ↑** |
| ③ 한계 초과 | 약간 ↑ | 크게 ↑ | **폭발 📈** |

→ 일부 요청이 큐에 쌓이면서 **지연이 폭발하기 시작한다.**

### 왜 이런 패턴이 나타나는가?

대부분의 서버는 내부적으로 **큐(Queue)** 기반으로 요청을 처리하기 때문에, 처리 한계를 넘으면 요청이 쌓이면서 대기 시간이 폭발적으로 증가한다.

```
1단계 초기      → 안정적인 응답
2단계 임계점    → 요청 대기 발생
3단계 한계 초과 → 지연 시간 급증
```

---

### 5-3. 요청 실패율 (Error Rate)

- 실패한 요청의 비율 또는 개수

### 체크해야 할 항목

- HTTP 5xx
- Timeout
- Connection 실패

### 핵심 의미

> **에러가 발생하는 시점 = 시스템 한계 도달**
>

## 6. 부하 테스트의 핵심 개념

부하 테스트의 목적은 단순한 수치 측정이 아니다.

> **시스템이 안정적으로 유지되는 최대 구간을 찾는 것**
> 

### 이상적인 흐름

1. 사용자 증가 → TPS 증가
2. 어느 순간 → TPS 정체 (Capacity 도달)
3. 이후 → Latency 증가 (특히 P99)
4. 더 이후 → Error 발생

→ 이 **전환 구간을 찾는 것**이 부하 테스트의 본질이다.

## 7. 테스트 환경 구성 원칙

### ❗ 부하 생성 서버는 반드시 분리

### 이유

- 부하 생성도 CPU/네트워크 자원을 사용한다
- 같은 서버에서 실행하면 결과가 왜곡된다

### 같은 서버에서 돌리면 생기는 문제

같은 서버에서 k6 + Spring Boot를 동시에 돌리면:

- k6가 CPU를 쓰는 만큼 Spring이 느려짐
- 측정된 Latency = **실제 처리 지연 + 부하 생성기의 자원 경합**
- 결과: "서버가 원래 느린 건지, k6 때문인지" 구분 불가

### 올바른 구조

```
[Load Generator]  →→→  [Target Server]
   (k6 실행)              (Spring Boot)
```

→ 순수한 서버 성능만 측정 가능, 결과 신뢰도 확보

대표적인 도구 3가지를 좀 더 깊이 비교해보자.

### 📊 한눈에 보는 비교표

| 항목 | **k6** | **JMeter** | **nGrinder** |
| --- | --- | --- | --- |
| 작성 방식 | JavaScript 코드 | GUI (XML 저장) | 웹 UI + Groovy 스크립트 |
| 기반 언어 | Go (런타임) | Java (JVM) | Java + Groovy |
| 실행 방식 | CLI | GUI / CLI | 웹 UI (Controller-Agent) |
| 메모리 사용량 | **VU당 ~1-5MB** (단순 테스트 기준) | 상대적으로 높음 | JVM 기반 (Agent 단위로 메모리 소비, 분산으로 완화) |
| 동시 사용자 처리 | 대규모 VU 
(스크립트와 장비 사양에 따라 달라짐) | 스레드 기반이라 상대적으로 제약 큼 | 대규모 VU
(Agent 분산으로 확장) |
| 동시성 모델 | Goroutine + Event Loop | Thread per VU
(VU 1명 = OS 스레드 1개) | Thread per VU
(Agent별로 분산 수용) |
| 분산 부하 | k6 Operator (Kubernetes) | JMeter 서버 분산 | **Controller-Agent 네이티브 지원** |
| 프로토콜 | HTTP, WebSocket, gRPC 중심 | 거의 모든 프로토콜 지원 | HTTP 중심 |
| CI/CD 연동 | CLI 친화적 | 별도 설정 필요 | 가능하지만 번거로움 |
| Git 협업 | 코드 기반 조작 | XML 기반 조작
변경 추적 어려움 | 코드기반 조작 가능
웹 UI에서 스크립트로도 조작가능 |
| 시각화 | Grafana + Prometheus | HTML 리포트 + 플러그인 | 웹 UI 내장 + CSV·tgz 다운로드 |
| 학습 곡선 | JS 알면 빠름 | GUI는 쉬움, 고급은 플러그인 의존 | 설정 복잡 (진입장벽) |
| 첫 출시 | 2017년 | **1998년** (역사 깊음) | 2011년 (네이버 사내 도구 → 오픈소스) |
- 단어정리
    
    ## Virtual User (VU)
    
    “가짜 사용자 한 명”
    
    - 실제 사람이 아니라
    - 사람이 하는 행동을 흉내 내는 존재
    - 로그인, 클릭, 조회 같은 행동을 반복함
    
    예: k6
    
    ---
    
    ## Thread (스레드)
    
     “컴퓨터가 일을 처리하는 작업자 1명”
    
    - 요청을 실제로 실행하는 단위
    - 1 Thread = 1 작업자
    
    예: Apache JMeter, nGrinder
    
    ---
    
    ## Virtual User vs Thread
    
    쉽게 보면
    
    - VU = 사람 역할
    - Thread = 일을 하는 몸
    
    ---
    
    ## Agent (에이전트)
    
    “부하를 대신 만들어주는 서버”
    
    - 여러 대를 사용해서 트래픽을 나눠서 보냄
    
    예:
    
    - nGrinder에서 사용

### 🔧 1. k6 - 요즘 많이 쓰인다

**핵심 철학: Performance as Code**

테스트 시나리오를 코드로 작성하고, 일반 코드처럼 Git으로 관리하며 코드 리뷰를 거친다는 철학.

**왜 가벼운가?**

Go 언어 런타임 기반으로 동작한다. JMeter처럼 VU 1명당 OS 스레드 1개를 그대로 띄우는 방식보다 대규모 테스트에 효율적인 구조를 가진다.

→ 단순 테스트 기준으로도 k6는 보통 **VU 1명당 약 1-5MB** 정도의 메모리를 사용할 수 있다.

→ 정확한 생성 가능 부하는 스크립트 복잡도, 응답 본문 처리 여부, 장비 사양에 따라 달라진다.

**장점**

- JavaScript(ES6+) 기반 코드 방식 → 백엔드/프론트엔드 개발자 모두 친숙
- Git 관리, 코드 리뷰, CI/CD 파이프라인 연동이 자연스러움
- 가볍고 빠름 (단일 인스턴스로 대량 VU 생성 가능)
- **Grafana + InfluxDB / Prometheus 연동**이 매우 강력 (Grafana Labs가 만든 도구)
- 실시간 메트릭 대시보드 제공

**단점**

- GUI 없음 (CLI 전용) → 비개발자 진입장벽
- HTTP/HTTPS, WebSocket, gRPC 중심 → JDBC나 SMTP 같은 프로토콜 지원 약함
- 커뮤니티가 JMeter에 비해 작음 (다만 빠르게 성장 중)

**언제 적합한가?**

- 개발팀이 직접 성능 테스트를 작성·관리할 때
- CI/CD 파이프라인에 부하 테스트를 통합하고 싶을 때
- HTTP API 위주의 모던 웹 서비스 (대부분의 경우)

### 🔧 2. JMeter — 레거시 기업 표준

**핵심 철학: GUI로 누구나 테스트 가능**

1998년부터 이어진 가장 오래된 부하 테스트 도구. "GUI에서 클릭하며 테스트를 구성한다"는 접근.

**JMeter의 강점은 광범위한 프로토콜 지원이다**

HTTP뿐 아니라 **JDBC(DB 직접), FTP, JMS(메시지 큐), SMTP, LDAP, SOAP** 등을 모두 지원한다. 다양한 시스템이 얽힌 엔터프라이즈 환경에서 강점.

**장점**

- GUI 클릭으로 쉽게 테스트 구성 → 비개발자(QA 팀)도 접근 가능
- 20년 이상 축적된 자료, 플러그인, 커뮤니티
- 다양한 프로토콜 지원 (DB, 메시지 큐, 메일 서버 등)
- 상세한 HTML 리포트 제공

**단점**

- **무거움**: JVM 기반과 스레드 모델 특성상 같은 부하 생성에 더 많은 자원이 필요할 수 있음
- **버전관리 어려움**: 테스트 플랜이 XML로 저장되어 Git에서 변경사항 추적 곤란. 코드 리뷰가 사실상 불가능 (변경 확인하려면 JMeter를 열어서 클릭해 확인해야 함)
- GUI는 디자인 용도이고 **실제 부하 생성은 CLI 모드 권장** ("Don't run load test using GUI mode!" — 공식 권고)
- 핵심 기능이 플러그인으로 분산되어 있어 신규 사용자가 어떤 플러그인을 깔아야 할지 모름

**언제 적합한가?**

- DB, 메시지 큐 등 **비-HTTP 프로토콜**까지 테스트해야 할 때
- QA 팀이 GUI 기반으로 작업하는 전통적 조직
- 이미 JMeter 자산(테스트 플랜)이 많이 쌓인 레거시 환경

### 🔧 3. nGrinder — 대규모 분산 테스트

**핵심 철학: 웹 UI에서 대규모 분산 부하**

네이버가 사내에서 쓰던 도구를 2011년 오픈소스로 공개. **Controller + Agent 구조**가 핵심 차별점.

**Controller-Agent 분산 구조란?**

```
[Controller (웹 UI)]
       │
       ├──→ [Agent 1] → 타겟 서버
       ├──→ [Agent 2] → 타겟 서버
       └──→ [Agent N] → 타겟 서버
```

여러 대의 Agent 머신에 부하 생성을 분산시킬 수 있어, **단일 서버로는 만들기 어려운 대규모 트래픽**을 만들 수 있다.

**장점**

- 분산 부하 테스트가 **네이티브로 지원** (k6는 별도로 k6 Operator + Kubernetes 필요)
- 웹 UI에서 테스트 실행 / 모니터링 / 결과 확인까지 한 곳에서
- 결과 리포트를 CSV, tgz로 다운로드 가능
- Groovy/Jython 스크립트로 시나리오 작성

**단점**

- **설정 복잡**: Controller, Agent를 각각 세팅해야 함
- 진입 장벽이 높음 (k6보다 처음 시작하기 어려움)
- 커뮤니티 규모는 k6/JMeter 대비 작음
- UI가 JMeter처럼 다소 클래식한 느낌

**언제 적합한가?**

- 단일 머신으로 만들 수 없는 **수십만 VU 이상**의 대규모 부하 필요할 때
- 사내에서 **공유 부하 테스트 플랫폼**을 구축하고 싶을 때 (여러 팀이 웹 UI로 접근)
- 네이버처럼 분산 환경에서 운영하는 대형 서비스

### 요약: 어느 도구를 언제 쓰나?

| 상황 | 추천 도구 |
| --- | --- |
| 일반적인 웹 API 부하 테스트 + CI/CD 자동화 | **k6** |
| DB·메일·메시지큐까지 테스트 + GUI로 작업 | **JMeter** |
| 대규모 분산 부하 + 사내 공용 테스트 플랫폼 | **nGrinder** |
| 개인 학습 / 사이드 프로젝트 | **k6** (가장 가볍고 빠르게 시작) |

## 9. 정말 "툴은 중요하지 않을까"?

절반만 맞는 말이다.

### ✅ 맞는 이유

- 핵심은 **지표 해석 능력**에 있다
- 어떤 도구든 TPS / Latency / Error Rate 를 읽고 해석할 수 있으면 목적은 달성된다

### ❌ 틀린 이유

- 협업, 자동화, 규모에 따라 도구의 영향이 크다
    - CI/CD 파이프라인 연동 가능 여부
    - 팀원과 스크립트 공유·버전관리
    - 대규모 분산 테스트 지원 여부

→ 결국 **지표를 읽는 눈 + 상황에 맞는 도구 선택** 이 진짜 실력이다.

## 9. 핵심 정리

> 성능 테스트는
> 
> 
> "서버가 언제 한계에 부딪히는지 보는 테스트"가 아니라
> 
> **"어디까지 안정적으로 처리할 수 있는지를 찾는 과정"이다**
> 

### 기억해야 할 3가지 지표

| 지표 | 의미 | 핵심 가치 |
| --- | --- | --- |
| **TPS** | 처리량 | 최대 처리 용량 파악 |
| **P99** | 지연 시간 (실제 사용자 경험) | 병목·장애 직전 감지 |
| **Error Rate** | 실패율 | 한계 초과 시점 감지 |
