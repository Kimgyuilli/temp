# 발표자용 스크립트

## Slide 01. 성능 테스트

오늘 발표는 성능 테스트 스터디 안에서, 그중 부하 테스트를 중심으로 다룹니다.

성능 테스트는 넓게 보면 시스템의 응답성, 안정성, 확장성을 확인하는 과정입니다. 오늘은 그중에서도 “예상되는 트래픽을 안정적으로 처리할 수 있는가”에 초점을 맞추겠습니다.

---

## Slide 02. 부하 테스트란?

부하 테스트는 시스템이 예상되는 트래픽을 안정적으로 처리할 수 있는지 검증하는 테스트입니다.

단순히 “몇 명까지 버티는가”를 보는 것이 아니라, 실제 운영 환경에서 문제없이 동작하는 범위를 찾는 과정이라고 보는 게 더 정확합니다.

---

## Slide 03. 왜 필요한가?

서비스 장애는 보통 하나의 문제로 끝나지 않습니다.

트래픽이 몰리면 처리 지연이 생기고, 지연이 길어지면 서비스 중단으로 이어질 수 있습니다. 그 결과 사용자가 이탈하고, 서비스 신뢰도도 같이 떨어집니다.

그래서 부하 테스트의 질문은 “서버가 터질까?”가 아니라 “어느 정도까지 안정적으로 운영할 수 있을까?”입니다.

---

## Slide 04. 실제 사례

수강신청과 티켓팅은 짧은 시간에 트래픽이 몰리는 대표적인 사례입니다.

수강신청은 매 학기 반복되는 피크 트래픽 문제이고, 티켓팅은 예매 시작 직후 몇 초에서 몇 분 사이에 트래픽이 집중됩니다.

예스24 사례는 부하 장애는 아니지만, 시스템이 멈추면 주문, 예매, eBook 같은 여러 서비스가 함께 영향을 받는다는 점을 보여주는 운영 리스크 사례로 보면 됩니다.

---

## Slide 05. 비용과 안정성

“그냥 서버를 크게 쓰면 되는 것 아닌가?”라는 질문이 나올 수 있습니다.

하지만 피크 시간 기준으로만 인프라를 잡으면 대부분의 시간에는 자원이 낭비됩니다. 반대로 너무 작게 잡으면 장애 위험이 커집니다.

부하 테스트는 이 사이에서 최소 비용으로 안정성을 확보할 수 있는 기준점을 찾기 위한 과정입니다.

---

## Slide 06. 테스트 유형

성능 테스트 안에는 여러 유형이 있습니다.

Load Test는 예상 트래픽에서 안정적으로 동작하는지 보는 테스트입니다. Stress Test는 한계를 넘겼을 때 어떻게 무너지는지 봅니다. Spike Test는 순간적인 폭증을 보고, Soak Test는 장시간 운영 중 성능 저하나 메모리 누수를 봅니다.

오늘 발표의 중심은 Load Test입니다.

---

## Slide 07. VU

VU는 Virtual User, 즉 가상 사용자입니다.

실제 사용자가 아니라, 사용자의 행동을 반복 실행하는 시뮬레이션 단위입니다. 예를 들어 VU 50명은 최대 50개의 사용자 시나리오가 동시에 실행되는 상황으로 볼 수 있습니다.

다만 도구마다 실행 모델은 다릅니다. k6는 VU를 런타임이 효율적으로 실행하고, JMeter와 nGrinder는 Thread 개념이 더 중요합니다.

여기서 Thread 기반 테스트와 VU 기반 테스트의 차이를 같이 이해하면 좋습니다.

Thread 기반 모델은 동시 사용자를 Thread 중심으로 표현합니다. 쉽게 말하면 사용자 1명을 작업자 Thread 1개에 가깝게 매핑해서 이해할 수 있습니다.

이 방식의 장점은 구조가 직관적이라는 점입니다. 각 사용자가 독립적인 작업자처럼 동작한다고 생각하기 쉽고, JMeter처럼 오래된 도구에서는 GUI, 플러그인, 다양한 프로토콜 지원 같은 생태계도 잘 갖춰져 있습니다.

다만 단점도 있습니다. 동시 사용자가 많아질수록 Thread 수가 늘어나고, 그만큼 메모리 사용량과 컨텍스트 스위칭 비용이 커질 수 있습니다. 그래서 아주 큰 규모의 동시성을 만들 때는 부하 생성기 자체가 무거워질 수 있습니다.

반대로 k6 같은 VU 기반 모델은 VU를 OS Thread와 1:1로 묶지 않습니다. VU는 실제 Thread라기보다는 사용자 행동을 표현하는 논리적인 단위이고, 실제 실행은 런타임이 효율적으로 처리합니다.

그래서 HTTP API 중심의 테스트에서는 VU 기반 모델이 더 유리한 경우가 많습니다. 같은 장비에서도 더 많은 가상 사용자를 만들기 쉽고, 테스트를 코드로 관리하기 좋고, CI/CD에 붙이기도 좋습니다.

여기서 중요한 점은, VU 모델 자체가 특정 프로토콜에 약한 것은 아니라는 점입니다. VU는 그냥 가상 사용자라는 추상화입니다. 어떤 프로토콜을 지원하느냐는 VU 모델의 한계라기보다, k6나 JMeter 같은 도구가 어떤 클라이언트와 플러그인을 제공하느냐의 문제입니다.

예를 들어 현재 생태계 기준으로는 k6가 HTTP, WebSocket, gRPC 같은 API 중심 테스트에 강하고, JMeter는 JDBC, JMS, SMTP, LDAP 같은 다양한 엔터프라이즈 프로토콜 지원에 강합니다.

그래서 VU 기반이 Thread 기반의 완전한 상위호환이라고 단정하기는 어렵습니다. 하지만 개념적으로는 VU가 Thread보다 사용자 행동에 가까운 더 높은 추상화라고 볼 수 있습니다.

만약 VU 기반 도구의 생태계가 더 발전해서 다양한 프로토콜, GUI, 분산 실행, 리포팅까지 충분히 갖추게 된다면, Thread 기반 도구가 하던 많은 역할을 대체할 가능성은 있습니다.

다만 기존 JMeter 생태계, 이미 쌓인 테스트 플랜, GUI 중심 QA 프로세스, 레거시 프로토콜 지원 같은 이유 때문에 Thread 기반 도구가 완전히 사라진다고 보기는 어렵습니다.

정리하면, HTTP API 중심의 현대적인 성능 테스트는 점점 VU 기반, 코드 기반 도구로 이동할 가능성이 큽니다. 반면 Thread 기반 도구는 레거시 환경, 다양한 프로토콜 테스트, GUI 중심 업무 방식에서 계속 쓰일 가능성이 높습니다.

## Slide 08. 핵심 지표

부하 테스트 결과를 볼 때는 크게 세 가지를 봅니다.

첫째는 TPS, 초당 처리량입니다. 둘째는 Latency, 특히 P95와 P99입니다. 셋째는 Error Rate, 실패율입니다.

이 세 가지를 같이 봐야 시스템이 어느 구간까지 안정적인지 판단할 수 있습니다.

---

## Slide 09. TPS

TPS는 초당 처리 가능한 요청 수입니다.

처음에는 VU가 늘어나면 TPS도 같이 늘어납니다. 그런데 어느 순간부터 VU를 더 늘려도 TPS가 증가하지 않는 구간이 나옵니다.

그 지점이 시스템의 처리 용량, 즉 Capacity에 가까운 지점입니다.

---

## Slide 10. Latency

Latency는 요청을 보낸 뒤 응답을 받을 때까지 걸리는 시간입니다.

여기서 평균만 보면 위험합니다. 대부분의 요청은 빠르지만 일부 요청만 매우 느린 경우, 평균은 괜찮아 보여도 실제 사용자 일부는 나쁜 경험을 하게 됩니다.

그래서 평균과 함께 P95, P99를 봐야 합니다.

---

## Slide 11. P99 변화

부하가 증가하면 평균보다 P99가 먼저 흔들리는 경우가 많습니다.

처리 한계에 가까워지면 일부 요청이 큐에 쌓이고, 그 요청들의 대기 시간이 길어집니다. 이때 평균은 아직 크게 변하지 않아도 P99는 빠르게 올라갈 수 있습니다.

P99가 급격히 튀기 시작하면 병목이나 장애 직전 신호로 봐야 합니다.

---

## Slide 12. Error Rate

Error Rate는 실패 요청의 비율입니다.

HTTP 5xx, Timeout, Connection 실패 같은 항목을 봅니다. 에러가 발생하기 시작했다는 것은 이미 안정 구간을 넘어섰다는 의미일 수 있습니다.

그래서 에러는 단순히 “실패가 몇 개 있었다”가 아니라, 시스템 한계에 가까워졌다는 신호로 해석해야 합니다.

---

## Slide 13. 전환 구간

부하 테스트의 본질은 전환 구간을 찾는 것입니다.

처음에는 VU가 늘면 TPS도 늘어납니다. 이후 TPS 증가가 멈추고, P99 Latency가 올라가기 시작합니다. 더 지나면 Error Rate가 증가합니다.

우리가 찾고 싶은 것은 장애가 난 지점이 아니라, 장애가 나기 전에 안정적으로 운영 가능한 마지막 구간입니다.

---

## Slide 14. 환경 구성

부하 생성 서버와 측정 대상 서버는 분리해야 합니다.

같은 서버에서 k6와 Spring Boot를 같이 실행하면, k6가 쓰는 CPU와 네트워크 자원이 측정 대상에도 영향을 줍니다.

그러면 서버가 느린 건지, 부하 생성기가 자원을 같이 써서 느려진 건지 구분하기 어렵습니다.

---

## Slide 15. 도구 비교

대표적인 도구로 k6, JMeter, nGrinder를 비교할 수 있습니다.

k6는 코드 기반이라 Git 관리와 CI/CD 연동에 강합니다. JMeter는 오래된 표준 도구이고 다양한 프로토콜을 지원합니다. nGrinder는 Controller-Agent 구조로 분산 부하 테스트에 강합니다.

도구 자체보다 중요한 것은 지표를 읽는 능력이지만, 팀의 환경과 테스트 규모에 따라 도구 선택도 결과에 영향을 줍니다.

---

## Slide 16. 언제 무엇을 쓰나?

일반적인 웹 API 부하 테스트와 자동화가 목적이면 k6가 가장 가볍게 시작하기 좋습니다.

DB, 메시지 큐, 메일 서버처럼 다양한 프로토콜까지 봐야 한다면 JMeter가 유리합니다.

여러 팀이 함께 쓰는 사내 부하 테스트 플랫폼이나 대규모 분산 테스트가 필요하면 nGrinder가 적합합니다.

---

## Slide 17. 핵심 정리

오늘의 핵심은 세 가지입니다.

TPS로 처리 용량을 보고, P99로 실제 사용자 경험과 병목 신호를 보고, Error Rate로 한계 초과 시점을 봅니다.

부하 테스트는 서버가 언제 터지는지 보는 테스트가 아니라, 어디까지 안정적으로 처리할 수 있는지 찾아가는 과정입니다.
