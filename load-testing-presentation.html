Section 03 · 테스트 유형

목적에 따라 4가지로 나뉜다.

종류 목적 활용 시점
Load Test 예상 트래픽에서 안정적으로 동작하는지 확인 이번 스터디의 주제
Stress Test 한계치 초과 시 시스템이 어떻게 무너지는지 내구 한계 파악
Spike Test 갑작스러운 트래픽 급증에도 안정적으로 응답하는지 순간 급증 대응 확인
Soak Test 장시간 운영 시 메모리 누수·성능 저하 발견 장기 안정성 검증

k6는 이 모든 유형을 options 시나리오 코드로 표현할 수 있다.

04 Virtual User

Section 04

VU. 가상 사용자.

VU = 사용자 행동을 반복 실행하는 시뮬레이션된 사용자 1명.
VU 50명 = 최대 50개의 사용자 시나리오가 동시에 실행되는 상황.

export const options = { vus: 50, // 가상 사용자 50명 duration: '30s' // 30초 동안 유지 };
VU
"가짜 사용자 한 명." 사람이 하는 행동을 흉내. k6
Thread
"일을 처리하는 작업자 1명." 1 Thread = 1 작업자. JMeter, nGrinder
Agent
"부하를 대신 만드는 서버." 분산 부하 생성. nGrinder

Section 05 · 핵심 지표

반드시 봐야 할
3가지 지표.

01
처리량TPS

초당 처리 가능한 요청 수. 시스템의 최대 처리 용량을 가늠하는 1차 지표.

02
지연 시간Latency

요청 → 응답까지 걸리는 시간. 평균이 아닌 P95·P99로 봐야 한다.

03
실패율Error Rate

실패 요청 비율. 에러가 발생하기 시작하는 지점이 시스템 한계의 신호다.

Section 05 — Metrics Overview 08 / 17
05·1 Throughput · TPS

처리량 · Transactions Per Second

① TPS.

TPS ≈ VU 수 ÷ (응답시간 + 대기시간) 예) VU=100, 응답시간=200ms, 대기시간=800ms → TPS ≈ 100 ÷ 1.0 = 100 TPS

실제 TPS는 VU 수, 응답시간, 대기시간, 시나리오 구조에 함께 영향을 받는다.

  • VU가 증가하면 TPS도 증가한다.
  • 그러나 어느 순간 증가가 멈춘다.
  • 이 지점이 → 시스템의 최대 처리 용량 (Capacity).
Section 05·1 — TPS 09 / 17
05·2 Latency · P95 / P99

지연 시간 · 평균의 함정

② Latency.
평균만 보면 왜곡된다.

요청 99개 → 100ms 요청 1개 → 3,000ms 평균 = 약 130ms 겉보기엔 빠른 서비스이지만,
일부 사용자는 느린 응답을 경험한다.
  • P95 — 100개 중 95개는 이 시간 이하. "대부분의 사용자 경험"
  • P99 — 100개 중 99개는 이 시간 이하. "최악에 가까운 경험" · 병목 감지
  • 실제 사용자 경험은 느린 요청이 좌우한다.
  • 평균은 "전체 속도", P95·P99는 "실제로 겪는 느림."
Section 05·2 — Latency 10 / 17
05·2 Load Increase Pattern

부하 증가 시 변화 패턴

P99는
평균보다 먼저 흔들린다.

구간 평균 P95 P99
① 초기 안정 안정 안정
② 임계점 도달 거의 유지 서서히 ↑ 급격히 ↑
③ 한계 초과 약간 ↑ 크게 ↑ 급격히 ↑↑

대부분의 서버는 내부적으로 큐(Queue) 기반으로 요청을 처리한다. 처리 한계를 넘으면 요청이 쌓이면서 대기 시간이 빠르게 증가한다.

05·3 Error Rate

실패율 · The Limit Signal

③ Error Rate.

핵심 의미

에러가 발생하는 시점은
시스템 한계의 신호.

반드시 체크할 항목.

  • HTTP 5xx — 서버 에러 응답
  • Timeout — 응답이 시간 내 안 옴
  • Connection 실패 — 연결 자체가 끊어짐

Section 06 · 핵심 개념

시스템이 안정적으로 유지되는
전환 구간을 찾는 것.

01

VU ↑ → TPS ↑

사용자가 늘어나면 처리량도 비례해서 증가하는 안정 구간.

02

TPS 정체

어느 순간 처리량 증가가 멈춘다 — Capacity 도달.

03

Latency ↑ (P99)

큐가 쌓이면서 지연 시간이, 특히 P99가 먼저 튀어오른다.

04

Error 발생

한계 초과. 에러 발생은 시스템이 더 이상 못 버틴다는 신호.

08 When to Use Each

어느 도구를 언제 쓰나?

상황에 맞는 도구 선택.

Performance as Code
k6

JS 기반 코드. Go 런타임 기반으로 대규모 테스트에 효율적. Grafana + Prometheus 연동이 강력. 단점은 GUI 없음, JDBC/SMTP 약함.

→ 일반적인 웹 API + CI/CD 자동화
Legacy Standard
JMeter

1998년부터 GUI로 누구나. JDBC·FTP·JMS·SMTP·LDAP 등 광범위한 프로토콜 지원. 단점은 무거움, XML로 Git 추적 곤란, 코드 리뷰 사실상 불가.

→ DB·메일·메시지큐 + GUI 작업
Distributed Native
nGrinder

네이버 사내 도구 → 2011 오픈소스. Controller + Agent 네이티브 분산 구조. 웹 UI에서 모니터링까지 한 곳. 단점은 진입 장벽 높음.

→ 수십만 VU 대규모 분산 + 사내 공용
